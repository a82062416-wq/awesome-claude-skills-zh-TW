## 3️⃣ Token 消耗優化（針對使用習慣）

使用者習慣：長對話、多輪迭代、財報類大檔。最燒 token 的三個源頭與對策：

| 源頭 | 對策 | 預期節省 |
|------|------|---------|
| MCP 工具回傳大量 JSON | CLI 優先（已立原則1）；MCP 只在無 CLI 替代時用 | 單次 5000→100 tokens |
| 大檔全讀進 context | 用 Grep/offset 只讀需要的段落，不整檔 Read；大檔交 subagent 只回結論 | 財報檔省 70%+ |
| 長對話歷史累積 | 換任務 `/clear`、長對話 `/compact`、研究派 researcher subagent | context 保持精簡 |

**落地**：token-budget-optimizer 技能（batch 2）把這些變成可查的檢查清單。
**鐵律**：回答前先問「這個資訊我需要全部，還是只要結論？」——要結論就派 subagent。

## 7️⃣ 產出一致性模板（缺失原因 + 強制機制）

**為什麼會缺**：技能是不同時間、不同來源（自建/fork/社群）累積的，沒有「新技能必須長這樣」的
單一範本與入口檢查，於是每個 SKILL.md 各長各的，README 與 marketplace 手動同步必然漂移。

**根因**：缺「產出時的強制關卡」——靠人記得同步，人必然忘。

**強制機制施工藍圖**（batch 3 落地）：
1. `template-skill/SKILL.template.md`：新技能唯一起點，欄位固定
2. `scripts/validate_repo.py` 擴充：技能存在但未註冊 marketplace / 未列 README → CI 擋下
3. skill-creator 產出時套用模板；validate 是最後防線
→ 機制設計原則：**不靠自律靠關卡**。忘了同步 = CI 紅 = 進不了 master。

## 8️⃣ 架構審計（耦合/控制單元/減法/讀寫不一致/死碼）

| 問題類型 | 本專案實例 | 根因 | 修補 |
|---------|-----------|------|------|
| **相似系統未融合（減法）** | 社區對帳 v1/v2/v3 三份 | 迭代時複製而非重構 | 停用 v1/v2，v3 唯一；未來改版用 git 分支不複製檔 |
| **寫入/讀出不一致** | 技能資料夾(41) ≠ marketplace(29) ≠ README | 三處手動維護 | validate 強制三者一致（batch 3） |
| **缺控制單元** | 41 技能無總表入口 | 沒設計索引 | skill-tree.html + consistency-checker 當控制單元 |
| **死碼** | SKILL.zh-TW.md 曾是死檔；marketplace 幽靈條目 | 迭代殘留 | validate 已抓；藍圖：定期掃孤兒檔 |
| **過度耦合** | hooks 硬編絕對路徑 /home/user/... | 圖方便 | 藍圖：路徑改相對或環境變數（🔴 需測試後改） |

**根因總結**：所有問題都源於「同一資訊存多處 + 靠人肉同步」。
**根治**：單一真實來源（Single Source of Truth）+ 機器強制一致性檢查。
本次 consistency-checker 是這個原則的第一個落地控制單元。

## 未來不再發生的自動化控制機制（總藍圖）

```
產出時   → SKILL 模板（唯一起點，欄位固定）
提交時   → validate_repo.py（技能↔marketplace↔README 三方一致、無死檔、無機密）
推送時   → GitHub Actions CI（同上，最後防線）
週期性   → 使用者手動 Routine 跑健康巡檢（技能健康度、記憶膨脹、孤兒檔）
制度變更 → 05-reflection.md 權限分級 + fresh-context 對抗審查
```

核心哲學：**每一種「靠人記得」的同步，都改成「機器強制」的關卡。**
