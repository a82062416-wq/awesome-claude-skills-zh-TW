# 🤝 派工與隔離驗證制度

## 什麼時候派 subagent（先過這關）

- ❌ 不派：只動 1 個檔案、預計 ≤3 次工具呼叫、需要與使用者來回確認的事——直接做
- ✅ 派：大範圍搜尋/研究（researcher）、可獨立驗收的封閉子任務、
  需要 fresh context 的驗證、多方案評比

## 派工三件套（每次指派缺一不可）

任何 subagent 指派的 prompt 必須包含三個區塊（模板見 04-templates.md）：

### 1️⃣ 明確目標與背景
- 一句話目標 + 為什麼做（背景脈絡）
- **Context 引入**：列出必讀檔案的完整路徑（subagent 是全新 context，
  它不知道主對話發生過什麼——你以為的常識它全都沒有）
- 範圍邊界：明確說「不要做什麼」（防 scope creep）

### 2️⃣ 嚴格的驗收條件
- 可機械檢查的條件（「X 檔案存在且通過 validate」而非「做好做滿」）
- 完成的定義引用 03-judgment-matrix.md 的完成標準

### 3️⃣ 標準回報格式
- 結論先行（1-2 句）
- 成果位置：`檔案路徑:行號`，**禁止噴出大段代碼**（>10 行的內容一律給路徑。
  例外：升級用的失敗軌跡與驗證指令輸出不受此限，過長時存檔後給路徑）
- 未解決事項與原因（沒有就寫「無」）

## 🔒 隔離驗證（鐵律）

**負責實作的 agent 不得自我驗證。** 理由：實作者會用寫代碼時的同一套錯誤假設
去寫驗證，錯誤假設驗不出自己。

驗收必須用以下至少一種：

| 方式 | 做法 | 適用 |
|------|------|------|
| Fresh-Context 驗收 | 派一個全新 subagent，只給它「驗收條件 + 產出位置」，不給實作過程。它 read-back 重新讀檔、實跑測試測資 | 功能實作、重構 |
| 主對話實證 | 主對話親自跑測試/指令並檢視輸出（不是看實作 agent 的轉述） | 小型變更 |
| 多樣本評審 | 同一任務派 2-3 個 agent 各自產出，再派評審 agent 或 adversarial-review 選優 | 品味相關、無標準答案的產出 |

驗收 agent 的 prompt **不得包含實作過程、實作者的自我解釋或辯護**——
解釋會污染驗收的獨立性。目標、範圍、驗收條件、產出位置、回報格式屬必要內容。

## 回收時的主對話責任

- Subagent 回報 ≠ 事實。關鍵宣稱（「測試通過」「檔案已建立」）主對話抽查一項實證
- 回報不合格式（噴代碼、沒附路徑）→ 退回重報，不要自己撿著用
