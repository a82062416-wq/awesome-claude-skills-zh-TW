# 🔄 知識迭代與反思協議

> 定義弱模型如何安全地更新這套 harness——哪些能自己改、哪些必須先問。
> 原則：**經驗層自由累積，制度層鎖定保護。**

## 修改權限分級

### 🟢 可自行更新（不必問，但要 commit 訊息說清楚）
| 檔案 | 規則 |
|------|------|
| `harness/lessons.md` | append-only 追加踩坑紀錄，不得刪改舊條目（經 🟡 核可的精簡程序除外） |
| `memory/MEMORY.md` | 照既有協定日常更新與汰舊（過時即刪屬 🟢） |
| `memory/plans/*`、`memory/archive/*` | 計畫與歸檔自由寫 |

### 🟡 可提案、不可直接改（把提案寫進 lessons.md 的「制度提案」段，等使用者核可）
| 檔案 | 理由 |
|------|------|
| **`harness/` 全目錄（lessons.md 除外，含本檔與 README）** | 制度是弱模型的護欄——護欄不能由被護的人自己拆，權限表本身也不例外 |
| `CLAUDE.md` | 全域行為定義，改壞影響所有 session |
| `fable-harness/`（協議與 hooks 腳本） | 行為協議層 |

### 🔴 必須使用者明確同意才能動
| 檔案 | 理由 |
|------|------|
| `.claude/settings.json`（hooks、permissions） | 改壞 = 自動化失控或安全洞 |
| `.github/workflows/` | CI 是最後防線 |
| `scripts/validate_repo.py` 的放寬修改 | 削弱驗證 = 判斷矩陣信號 2 的制度版。加嚴可提案，放寬必問 |
| 刪除任何制度檔案 | — |

## 踩坑紀錄格式（lessons.md）

一坑一行，固定格式：

```
- YYYY-MM-DD｜情境（10字內）｜錯誤做法 → 正確做法｜一句話規則
```

範例：
```
- 2026-07-07｜技能載入｜用 SKILL.zh-TW.md 命名 → 必須有 SKILL.md｜載入器只認標準檔名，別名無效
```

寫入時機：每次「重試才成功」「被驗證閘/CI 擋下」「使用者糾正」之後，立即寫。

## 精簡與抽象化觸發（防記憶腐化）

兩級門檻，兩種程序：

- **MEMORY.md 日常汰舊 = 🟢**：150 行為軟觸發（該瘦身了）、200 行為硬上限
  （validate_repo.py 會警告）。刪過時條目、合併重複——自己做，不必問
- **lessons.md 批次精簡 = 🟡**：超過 **80 條** 觸發，屬提案級操作：
  1. 同類坑合併成一條抽象規則（保留規則，丟棄故事）
  2. 已寫進制度檔（01~04）的坑，從 lessons 移到 `memory/archive/lessons-YYYY.md`
  3. 產出精簡後版本 + 一句話說明刪了什麼，請使用者過目後才落檔
- 抽象化的品質標準：新規則必須讓「沒看過原始故事的模型」也能執行。
  寫完自問：Haiku 讀得懂並照做嗎？讀不懂就重寫

## 反思節奏

- 每次中大型任務收尾（/retro）：教訓 → lessons.md，決策 → MEMORY.md
- 使用者手動觸發的每週巡檢：順帶檢查 lessons/MEMORY 是否到達精簡閾值
