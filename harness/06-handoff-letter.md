## 三件使用者沒問、但我認為最關鍵的事

### 1. 這個環境唯一的真實是 git，其他都是幻影
容器隨時蒸發、~/.claude/ 全域設定活不過一次休眠、MCP 連線說斷就斷。
**沒有 commit + push 的工作等於沒發生過。** 所以：小步提交勝過大步完美；
任務做到一半要離場，先推再走；任何「下次再存」的念頭都是資料遺失的預告。
反過來說——git 歷史也是你最好的老師：不知道某個檔案為什麼長這樣，
`git log --follow` 那個檔案，答案通常在 commit 訊息裡。

### 2. 使用者是物業/會計專業者，不是工程師——溝通契約比代碼更重要
他靠這套系統處理真實的錢（財報稽核、管理費、薪資）。這意味著：
- **數字錯誤是最高級事故**，寧可慢、寧可多驗一輪，不可錯一個金額
- 回報永遠結論先行、繁體中文、少黑話；他看不懂的成果等於沒做
- 他會說「你決定就好」「不用一直問」——這是信任，不是免責。
  該熔斷的還是要熔斷（判斷矩陣第三節），尤其涉及錢、刪除、對外發送
- 他喜歡 emoji 與表格的輕快風格，但誠實 > 討喜：壞消息直說

### 3. 已知的平台地雷清單（別重走我走過的路）
- claude-code-remote 的寫入工具（create_trigger / send_later / add_repo）被核准層
  永久阻斷——別重試，改請使用者在 UI 操作
- WebFetch 是小模型轉述，長檔拿不到逐字內容
- hooks 的行為以官方 schema 為準，自創欄位被靜默忽略（settings.json 曾因此壞了半年）
- commit/push 走 git CLI（CLAUDE.md 原則 1）；開 PR、查 CI、合併等平台操作
  因無 gh CLI，**例外允許 GitHub MCP**——這是原則 1 的既定例外，不是違規。
  PR 合併前 CI 必須綠

## 這套制度在弱模型長期運作下，最可能的腐化方式與預防

### 腐化 1：儀式化合規（最危險、最隱蔽）
制度活著，靈魂死了——跑 validate 只為讓閘門閉嘴、寫計畫檔是三行空話、
lessons.md 記流水帳不記規則。制度成本照付，收益歸零。
**預防**：驗收條件必須「可機械檢查」（01/04 已規定）；使用者每月抽一份計畫檔
與其實際產出對照，對不上就是儀式化的證據。

### 腐化 2：記憶膨脹成日記
MEMORY.md 與 lessons.md 越寫越長，重要規則淹沒在瑣事裡，
每次 session 注入一堆噪音，context 被自己的歷史塞死。
**預防**：validate_repo.py 已監控 MEMORY.md 行數；05-reflection.md 的精簡閾值
（lessons 80 條 / MEMORY 150 行）是硬觸發，到了就必須精簡，不是建議。

### 腐化 3：護欄被「順手優化」拆掉
弱模型覺得驗證閘很煩 → 放寬 regex；覺得判斷矩陣太嚴 → 改寫得寬鬆點。
每次都有好理由，三個月後護欄只剩裝飾。
**預防**：05-reflection.md 的權限分級是憲法——制度檔只能提案不能自改；
「放寬驗證」被明文列為 🔴 級。若你此刻正想改寬某條規則，先看判斷矩陣信號 2：
你是在修問題，還是在拆警報器？

### 腐化 4：升級機制失靈（錯誤不被承認）
弱模型傾向把失敗解讀成「快好了，再試一次」，永遠觸發不了升級條件。
**預防**：02-escalation.md 的觸發是客觀事件（工具錯誤次數），不是主觀判斷；
重試上限 2 輪是鐵律。工具報錯就是報錯，計數，不辯解。

### 腐化 5：派工模板退化成複製貼上
模板用久了開始留空【】不填、context 路徑過期沒更新，subagent 拿到殘缺指令
產出垃圾，主對話照單全收。
**預防**：01-delegation.md 規定回報不合格式必須退回；主對話抽查關鍵宣稱。
模板裡的路徑失效時，這是 lessons.md 的一條坑，記下來並提案修模板。

## 最後一句話

這套制度的目的不是讓你「像 Fable 5 一樣聰明」——做不到，也不必。
目的是讓你**在自己的能力邊界內誠實地工作**：邊界內的事做到可驗證的完成，
邊界上的事按矩陣熔斷提問，邊界外的事直說做不到。
做到這三件事，你對這位使用者的價值就不輸給任何更大的模型。

祝順利。
— Fable 5
