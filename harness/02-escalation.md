# 🪜 異常升降級路徑（模型分工與熔斷）

## 模型分工基準

| 模型 | 用途 | 不要給它 |
|------|------|---------|
| Haiku | 批次套用已定型的模式、格式轉換、簡單抽取 | 任何需要判斷的事 |
| Sonnet | 日常實作、搜尋研究、標準流程執行 | 無前例的架構決策 |
| Opus 4.8 | 疑難排錯、架構決策、被升級上來的難題 | 大量重複性批次工作（浪費） |

## 升級規則（客觀觸發，不靠自覺）

```
Haiku 出現 1 次工具錯誤或語法錯誤
  → 直接升級 Sonnet（不重試。小模型的錯誤是能力信號，不是運氣）

Sonnet 同一子任務連續失敗 2 次
  → 升級 Opus 4.8，且必須帶上【完整失敗軌跡】：
     - 兩次分別嘗試了什麼（指令/diff）
     - 兩次分別怎麼失敗（完整錯誤輸出，不是轉述）
     - 已排除的假設
  ⚠️ 沒帶失敗軌跡的升級無效——Opus 會重走一遍你走過的死路

Opus 4.8 解出固定模式後
  → 把模式寫成明確規則/範本（存入 harness/lessons.md 或專用範本檔）
  → 降回便宜模型批次套用。Opus 只解「第一次」，不解「每一次」
```

## 重試上限（鐵律）

- **同一件事、同一種做法，最多重試 2 輪。** 第 2 次失敗後，禁止第 3 次原樣重試
- 第 3 步只有三個合法選項：**換路徑**（不同做法）、**升級**（帶失敗軌跡）、
  **熔斷提問**（見 03-judgment-matrix.md）
- 「多試一次說不定就好了」是弱模型最貴的幻覺——環境性錯誤（網路、暫時鎖）
  例外，但重試前必須說出「我判斷這是環境性錯誤，因為＿＿」

## 🔌 Context/Token 熔斷

當 context window 明顯進入尾聲（回應變慢、內容被壓縮、任務只剩零頭卻堆著大量歷史）：

1. **立刻停止產出新內容**
2. 依序完成：① 已完成的工作 commit + push；② 未完成項目與下一步寫進
   memory/MEMORY.md 或交接檔；③ 在主對話留一頁摘要
3. 寧可少做一項任務，不可留下「做了一半且沒人知道做到哪」的狀態
