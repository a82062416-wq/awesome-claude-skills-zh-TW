# 📨 標準化派工 Prompt 模板

> 用法：複製對應模板 → 填空【】→ 派工。三件套缺一不可（見 01-delegation.md）。
> 回收時：回報不合格式就退回重報。

---

## 模板 1：搜尋研究（researcher agent）

```
【目標】查清：＿＿＿。這是為了＿＿＿（背景一句話）。
【必讀 context】先讀這些檔案再開始：
- /home/user/awesome-claude-skills-zh-TW/【相關檔案路徑】
【範圍】只查＿＿＿；不要研究＿＿＿；網路來源優先官方文件與 2025 年後的資料。
【驗收條件】
1. 回答涵蓋：【問題點1】【問題點2】
2. 每個結論附來源（檔案路徑:行號 或 URL）
3. 查不到的部分明確標註「查無」，不得推測填充
【回報格式】結論先行（≤3句）→ 證據條列（每條一行）→ 建議（如適用）。
禁止貼大段原文；超過 10 行的內容給路徑或連結。
```

## 模板 2：功能實作

```
【目標】實作：＿＿＿。使用者需求原文：「＿＿＿」。
【必讀 context】
- 規格/計畫：memory/plans/【計畫檔】
- 相關現有代碼：【路徑】（先讀懂再動手，模仿既有風格）
- 專案規範：CLAUDE.md、harness/03-judgment-matrix.md 的完成條件
【範圍】只改【目錄/檔案清單】；不要動【明確排除項】；發現需要動範圍外的東西→停下回報。
【驗收條件】
1. 【可機械檢查的行為條件，例：執行 X 指令輸出 Y】
2. python3 scripts/validate_repo.py 通過
3. 改功能邏輯必附 fail-then-pass 證據
4. 新代碼風格與周圍代碼一致（註解密度、命名）
【回報格式】結論先行 → 變更清單（檔案路徑:行號 + 各一句話說明）→
驗證輸出（貼指令與結果）→ 未解決事項（無則寫「無」）。禁止在回報中貼大段代碼。
【升級規則】同一障礙失敗 2 次 → 停止，輸出完整失敗軌跡（嘗試/錯誤輸出/已排除假設）。
```

## 模板 3：代碼重構

```
【目標】重構：＿＿＿。動機：＿＿＿（可讀性/重複/效能——只選真正的動機）。
【必讀 context】待重構代碼：【路徑】；呼叫它的地方：先 grep 找出完整呼叫面。
【範圍鐵律】行為不得改變——重構 = 結構變、行為不變。發現 bug 不要順手修，
記下來回報（修 bug 是另一個任務）。
【驗收條件】
1. 重構前先記錄基準行為：【測試輸出/指令結果】
2. 重構後同一驗證產出相同結果（貼前後對照）
3. 所有呼叫點都已更新，grep 舊符號 = 0 結果
4. python3 scripts/validate_repo.py 通過
【回報格式】結論 → 結構變更摘要（3 句內）→ 前後行為對照證據 →
呼叫點更新清單（路徑:行號）。
```

## 模板 4：代碼審查（fresh-context 驗收）

```
【目標】獨立驗收以下產出，你沒有參與實作，用全新眼光審。
【產出位置】【檔案路徑清單】（自己 read-back，不要相信任何人的轉述）
【驗收條件】逐條判定 PASS/FAIL 並附證據：
1. 【原派工的驗收條件，原樣複製過來】
2. 實跑：【驗證指令】，貼輸出
3. 對照 harness/03-judgment-matrix.md 第二節的完成條件表
【範圍】只驗收，不要修。發現問題列出來，由主對話決定退回或修正。
【回報格式】總判定（PASS/FAIL）→ 逐條結果表 → FAIL 項的具體位置（路徑:行號）
與缺什麼。品味類意見標註「建議」與必要條件分開。
```

---

## ⚠️ 通用注意

- Subagent 是全新 context：主對話的任何共識、縮寫、代號它都不知道，模板裡寫全
- 每個模板的【必讀 context】用完整絕對路徑
- 派工後回收時，主對話至少抽查一項關鍵宣稱（見 01-delegation.md）
