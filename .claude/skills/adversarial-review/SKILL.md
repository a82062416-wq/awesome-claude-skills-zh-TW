# 多方抗辯審查（Adversarial Review）

## 觸發條件

- 使用者明說：「抗辯」「挑戰這個結論」「對抗審查」
- 自動觸發：架構決策、根因判定、安全判斷、會影響生產/正式資料的變更

## 執行規則

1. **同一則訊息平行派出三個反方**（用 Agent 工具、一次三個，禁止串行）：
   - `skeptic`（質疑者）：找邏輯漏洞、未驗證的假設、反例
   - `red-team`（紅隊）：找安全漏洞、邊界條件、注入面、失敗模式
   - `simplifier`（簡化者）：質疑過度工程，找更簡單的替代方案
2. **禁止合併三個鏡頭為單一 agent**——獨立性就是價值所在
3. 每個反方回傳 `verdict: REFUTED | SURVIVED` + 理由

## 裁決（過半存活制）

| 存活票數 | 處置 |
|---------|------|
| 3/3 | 直接採信 |
| 2/3 | 採信，但 REFUTED 方的理由列入風險清單一併回報 |
| ≤1/3 | 結論擋回，修正後重審 |

- 生產影響級結論：修正後重審，直到**連續兩輪無新反駁**才算完工
- REFUTED 的理由不得隱匿，必須完整回報給使用者

## 回報格式

表格呈現：每個鏡頭一列（判決 + 關鍵理由），最後一行總結
「N/3 存活 → confirmed / 擋回修正」。
